# 小田急電鉄 データ確認リスト
## 小田急 沿線観光インテリジェンス PoC

作成：Tradfit × Cellmuller  
対象：小田急電鉄 観光・交通・商業・デジタル担当部門  
目的：PoCで使用するデータの保有可否・粒度・提供条件を確認する

---

> **確認の進め方について**
> 全てのデータが揃わなくても開始できます。
> 「A：鉄道/バス乗降」「B：フリーパス」の2種類があれば初期PoC（回遊ファネル・箱根マップ）は成立します。
> 各項目に「◎ある」「△一部ある」「✗ない」「？不明」でご回答いただけると助かります。

---

## PART 1｜必須データ（これがないとPoC成立しない）

---

### A. 鉄道乗降データ

**何に使うか**：箱根エリアへの来訪集中度・回遊ファネルの土台になります。どの駅に何人来たかを時間帯別に見ることで「大湧谷に集中している」「仙石原は余地がある」が分かります。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **保有可否** | 駅別の乗降集計データは存在しますか？ | ◎ / △ / ✗ / ？ |
| **対象路線** | 小田急線・箱根登山鉄道・ケーブルカー・ロープウェイそれぞれありますか？ | |
| **時間帯粒度** | 1時間単位で取れますか？（最低でも午前/午後/夕方の3区分あれば可） | |
| **日次更新** | 日次で取れますか？月次集計のみですか？ | |
| **エリア紐づけ** | 駅コード/駅名と「箱根湯本」「大涌谷」等のエリア名を紐づけるマスタはありますか？ | |
| **データ形式** | CSVで出力できますか？他のシステムからAPIで取れますか？ | |
| **管轄部署** | このデータを持っている部署はどこですか？ | |
| **提供可能期間** | 何年何月から取れますか？（最低1年分あると季節比較ができます） | |
| **個人情報有無** | 集計データのみで個人IDは含まれていませんか？ | |

**最低限必要なカラム**
```
data_date（日付）/ hour（時間）/ station_name（駅名）/ boardings（乗車数）/ alightings（降車数）
```

---

### B. バス乗降データ（箱根登山バス等）

**何に使うか**：鉄道では見えない「箱根湯本→大湧谷」「箱根湯本→仙石原」の移動ルートの分布が分かります。どのルートに集中しているかを見るために必要です。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **保有可否** | 停留所別の乗降集計データは存在しますか？ | ◎ / △ / ✗ / ？ |
| **対象路線** | 箱根登山バス・小田急箱根高速バス等、どの路線がありますか？ | |
| **時間帯粒度** | 1時間単位または便単位で取れますか？ | |
| **ルート情報** | バス路線ID（例：T路線、K路線）と停留所の紐づけはありますか？ | |
| **停留所→エリア変換** | 停留所名から「大涌谷エリア」等に変換するマスタはありますか？ | |
| **管轄部署** | 箱根登山バスのデータを小田急電鉄として取得できますか？グループ会社との調整が必要ですか？ | |
| **データ形式** | CSVで出力できますか？ | |

**最低限必要なカラム**
```
data_date / hour / stop_name（停留所名）/ route_id（路線） / boardings / alightings
```

---

### C. エリア・駅・停留所マスタ

**何に使うか**：乗降データをどのエリアに分類するかを決める辞書です。これがないと「箱根湯本」「大涌谷」「仙石原」等のエリア単位で集計できません。PoC開始前に最初に必要なものです。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **保有可否** | 駅・停留所と観光エリアを紐づけたマスタはありますか？ | ◎ / △ / ✗ / ？ |
| **エリア定義** | 「箱根湯本」「大涌谷」「強羅」「仙石原」「小田原」等のエリア区分は社内で定義されていますか？ | |
| **緯度経度** | 各駅・停留所の位置情報（緯度経度）はありますか？（地図表示に使います） | |
| **対象施設の網羅性** | 提携施設（美術館・温泉・土産店等）はこのマスタに入っていますか？ | |

## PART 2｜推奨データ（あると分析の精度が大きく上がる）

---

### D. 箱根フリーパス・券種別販売・利用データ

**何に使うか**：フリーパスを持っている人は「複数エリアを回ろうとしている観光目的の人」として分類できます。購入場所・利用エリアを見ることで「どの層がどこに行っているか」のセグメント分析が可能になります。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **保有可否** | フリーパスの販売・利用集計データはありますか？ | ◎ / △ / ✗ / ？ |
| **券種の種類** | 箱根フリーパス2日・3日、子ども用、外国語版等、どの券種がありますか？ | |
| **販売チャネル** | アプリ・Web・駅窓口・旅行代理店等、チャネル別に分かりますか？ | |
| **利用エリア** | どのエリアで使われたか（改札・乗降・施設入館）が分かりますか？ | |
| **日次粒度** | 日次で取れますか？ | |
| **国籍・言語** | 外国語版の販売比率等、インバウンド判別に使える情報はありますか？ | |
| **管轄部署** | チケット管理部門はどこですか？ | |

**最低限必要なカラム**
```
data_date / ticket_type（券種）/ sales_channel（販売チャネル）/ usage_area（利用エリア）/ usage_count（利用数）
```

---

### E. Web・アプリ閲覧データ

**何に使うか**：「Webで仙石原を検索しているのに現地には来ていない」というギャップが分かります。「行きたいが行けていない場所」を発見し、そこへの誘導施策の優先順位を決める材料になります。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **Webサイト計測** | 小田急観光サイト・箱根観光サイトにGA4等の計測は入っていますか？ | ◎ / △ / ✗ / ？ |
| **アプリ計測** | 小田急アプリのページ閲覧・検索ログは取れていますか？ | |
| **エリア別集計** | 「箱根湯本」「仙石原」等のエリアページ別の閲覧数が取れますか？ | |
| **言語別集計** | 日本語・英語・中国語・韓国語等、言語別に取れますか？（インバウンド分析に使います） | |
| **検索キーワード** | サイト内検索のキーワードログはありますか？ | |
| **コンバージョン** | クーポン保存・チケット購入・ルート保存等のイベントは計測されていますか？ | |
| **BigQuery連携** | GA4のデータをBigQueryにエクスポートしていますか？（大量データの分析に必要です） | |
| **データ保持期間** | 過去何ヶ月分のデータが残っていますか？ | |

**最低限必要なカラム**
```
data_date / channel（web or app）/ page_category（エリア・ルート・施設等）/ page_area_id / language / views / clicks
```

---

### F. 商業施設・購買データ

**何に使うか**：「大湧谷に1万人来ているのに購買転換率が18%しかない」という消費機会ロスを可視化するために使います。乗降データと掛け合わせることで「人はいるのにお金が落ちていない場所」が特定できます。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **保有可否** | グループの商業施設（土産店・飲食・温泉等）の購買集計データはありますか？ | ◎ / △ / ✗ / ？ |
| **対象施設の範囲** | どの施設のデータが取れますか？（小田急系列のみ？提携施設も？） | |
| **実金額の提供可否** | 実際の売上金額は出せますか？難しい場合は「売上指数」（実額を出さない形）でも可です | |
| **施設カテゴリ** | 飲食・物販・温浴・文化施設等のカテゴリ分けはされていますか？ | |
| **時間帯粒度** | 1時間単位で取れますか？ | |
| **クーポン利用** | クーポン利用数が施設・日次で取れますか？ | |
| **管轄部署** | 商業施設のデータを持っている部署はどこですか？（交通部門とは別の部署ですか？） | |

**最低限必要なカラム**
```
data_date / facility_id（施設ID）/ facility_category（カテゴリ）/ transaction_count（件数）/ sales_amount_index（売上指数）
```

---

### G. 施設マスタ（商業施設・提携施設の一覧）

**何に使うか**：消費分析の表示軸になります。「どの施設が飲食で、どの施設が温浴か」を分類するためのマスタです。購買データと組み合わせて「エリアごとの消費カテゴリ傾向」を出します。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **保有可否** | グループ・提携施設の一覧（施設名・カテゴリ・所在エリア）はありますか？ | ◎ / △ / ✗ / ？ |
| **カテゴリ定義** | restaurant / museum / shop / hot_spring / hotel 等に分類されていますか？ | |
| **位置情報** | 施設の緯度経度はありますか？（地図上の表示に使います） | |
| **提携施設の範囲** | 小田急グループ外の提携施設（民間温泉・飲食店等）も含まれますか？ | |

## PART 3｜施策効果測定に必要なデータ

---

### H. クーポン・キャンペーン・QR反応データ

**何に使うか**：「仙石原へのQRコード誘導施策を打ったあと、仙石原の乗降数が増えたか」を自動で計測するために使います。施策IDと紐づけることで「この施策は効いた/効いていない」が数字で分かります。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **クーポン管理** | 発行したクーポンの利用数・利用場所・利用日が取れますか？ | ◎ / △ / ✗ / ？ |
| **QRコード計測** | QRスキャン数が施策・場所・日次で取れますか？ | |
| **施策ID管理** | 各施策（QR・クーポン・Web・サイネージ）に共通のIDは付いていますか？（これがないと前後比較ができません） | |
| **施策の開始・終了日** | どの施策をいつからいつまで実施したかの記録はありますか？ | |
| **対象エリア** | 各施策がどのエリアを対象にしていたか分かりますか？ | |
| **チャネル種別** | QR・クーポン・Web・アプリプッシュ・サイネージ等、チャネル別に分かりますか？ | |
| **管轄部署** | 施策データを管理している部署はどこですか？（マーケティング部門？現場担当？） | |

**最低限必要なカラム**
```
campaign_id（施策ID）/ data_date / area_id / channel / impressions（表示数）/ scans（スキャン数）/ redemptions（利用数）
```

> **重要**：施策IDが存在しない場合、効果測定画面は作れません。PoC開始前に施策IDの設計だけでも先行して合意させてください。

## PART 4｜精度向上データ（Phase 2以降でも可）

---

### I. ロマンスカー予約・利用データ

**何に使うか**：ロマンスカー利用者は事前に計画して来る層であり、高付加価値旅行者の傾向を持ちます。「来月のロマンスカー予約が増えているエリア」から富裕層向け商品の商機を事前に察知できます。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **保有可否** | ロマンスカーの予約集計データはありますか？ | ◎ / △ / ✗ / ？ |
| **粒度** | 乗車日×発着駅×時間帯×席種で取れますか？ | |
| **予約リードタイム** | 何日前に予約されたかの情報はありますか？（事前需要の先読みに使います） | |
| **人数構成** | 1名・2名・グループ等の人数分布は取れますか？ | |

---

### J. デジタルサイネージ接触データ

**何に使うか**：箱根エリアのサイネージで「仙石原へどうぞ」と案内した後、実際に仙石原への乗降が増えたかを測るために使います。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **設置場所** | デジタルサイネージはどの駅・施設に設置されていますか？ | ◎ / △ / ✗ / ？ |
| **表示ログ** | どの施策コンテンツがいつ何回表示されたかのログはありますか？ | |
| **管理システム** | サイネージ管理システムからCSVやAPIでデータを取れますか？ | |

---

### K. 天気・イベントデータ

**何に使うか**：「雨の日は大湧谷への乗降が30%減る」という天候補正をかけることで、施策の効果を純粋に評価できます。また「花火大会の翌週に仙石原の来訪が増えた」等のイベント効果も補正できます。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **天気データ** | 箱根エリアの日次天気データを社内で保持していますか？（なければ気象庁のオープンデータで補完可） | ◎ / △ / ✗ / ？ |
| **イベント管理** | 箱根・沿線で実施するイベント（花火・祭り・キャンペーン等）の日程一覧はありますか？ | |
| **イベントのエリア紐づけ** | 各イベントがどのエリアで開催されるか分かりますか？ | |

## PART 5｜将来拡張データ（Phase 3以降）

以下は初期PoCでは不要ですが、将来的に確認しておきたい項目です。

| データ | 用途 | 確認事項 |
|---|---|---|
| **外国人旅行センター接客ログ** | 定性ニーズの補助。国籍・相談カテゴリ等 | 相談内容を記録するシステム・フォームはありますか？ |

## PART 6｜部門横断の調整確認

データ収集で最も難しいのは「技術的な問題」より「部門間の調整」です。以下を確認させてください。

| 確認項目 | 内容 | 回答欄 |
|---|---|---|
| **窓口担当者** | 今回のPoCのデータ取りまとめ窓口は誰になりますか？ | |
| **交通データの取得権限** | 鉄道・バス乗降データは観光担当部門が直接取れますか？それとも交通企画部門への依頼が必要ですか？ | |
| **商業施設データの取得権限** | ショッピングセンター購買データは観光部門が取れますか？別部署との調整が必要ですか？ | |
| **Webデータのアクセス権** | GA4・アプリログへのアクセス権はありますか？デジタル部門との連携が必要ですか？ | |
| **グループ会社との連携** | 箱根登山鉄道・箱根登山バスのデータを小田急電鉄が使うための社内・グループ間の手続きはありますか？ | |
| **個人情報審査** | 集計データの外部提供（Cellmuller・Tradfit）について、法務・コンプライアンス部門の承認が必要ですか？ | |
| **データ提供の形式** | CSVファイル共有（S3へのアップロード等）、API連携、どちらが御社にとって現実的ですか？ | |

## 優先度サマリー

| 優先度 | データ | なければどうなるか |
|---|---|---|
| **必須** | A. 鉄道乗降 | 回遊ファネル・来訪集中度が作れない |
| **必須** | C. エリア・駅マスタ | すべてのデータをエリアに紐づけられない |
| **必須** | 施策ID設計（H） | 効果測定画面が成立しない |
| **推奨** | B. バス乗降 | ルート別の分散分析の精度が落ちる |
| **推奨** | D. フリーパス | 観光目的セグメントが識別できない |
| **推奨** | E. Web/アプリ | 「行きたいが来ていない」ギャップが見えない |
| **推奨** | F. 商業施設購買 | 消費機会ロスの算出ができない |
| **任意** | I. ロマンスカー | 高付加価値層の傾向が見えない |
| **任意** | J. サイネージ | サイネージ施策の効果測定ができない |
| **任意** | K. 天気・イベント | 変動要因の補正ができない |
